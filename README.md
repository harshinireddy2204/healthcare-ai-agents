### How the demo works

The deployment splits features by **cost**: anything that reads data, queries a free API, or shows pre-recorded output is **open to everyone**. The two things that consume paid OpenAI tokens are **gated by passcode** so the shared API budget is not drained by traffic.

### Architectural rationale

This is not a limitation — it is the right production tradeoff. At any real health system, you would separate compute-heavy agent execution from read-only dashboard traffic anyway. The cached-run pattern is how any expensive AI system (pathology image classifiers, genomic workflows, radiology AI) gets demo'd at scale.

## What Problem This Solves

Clinical operations teams at health systems spend 30–40% of administrative time on:
- **Prior authorization** — manually verifying whether treatments meet payer coverage criteria
- **Care gap detection** — identifying patients missing preventive measures (mammograms, HbA1c checks, colonoscopies)
- **Risk triage** — prioritizing high-risk patients for proactive outreach before adverse events

This system automates all three using specialized AI agents grounded in real clinical guidelines, real FDA drug data, and a knowledge graph of 200+ evidence-based clinical relationships — with every decision audited and escalated to clinicians when confidence is low.

## Key Features

### Adaptive Complexity Routing (MDAgents pattern)
Before any agent runs, a complexity scorer evaluates the patient using diagnosis count, high-risk medications, lab values, and knowledge graph risk tier. P019 (Migraine + Anxiety) takes the fast single-agent path. P004 (Heart Failure + AFib + CKD Stage 4 + Diabetes + Warfarin) gets the full ICT with drug safety analysis. This reduces token cost by ~60% on routine cases.

### Real-Time Drug Safety (TxAgent / MALADE pattern)
Calls the OpenFDA API live for each patient's medication list — fetches actual FDA drug labels and extracts `drug_interactions`, `contraindications`, and `boxed_warnings` sections. Caught P004's Metformin contraindication in CKD Stage 4 (FDA: lactic acidosis risk) and Warfarin + NSAIDs bleeding risk.

### Agent/Critic Prior Auth (MALADE pattern)
After the primary ReAct agent makes a prior auth decision, a dedicated Critic agent reviews the reasoning for logical gaps. Only decisions that survive critic review are finalized — everything else gets one revision cycle before the HITL queue.

### RAG Over Clinical Guidelines
ChromaDB vector store with 63 guideline sources: USPSTF, ADA 2025, ACC/AHA, KDIGO 2024, NCI PDQ, CDC immunization schedules, NICE mental health, NIAMS rheumatology, and more. Uses OpenAI's `text-embedding-3-small` for vector embeddings + clinical synonym expansion (diabetes -> diabetic, CKD -> chronic kidney disease). Care gap outputs cite the specific guideline: *"Mammogram overdue — USPSTF Grade B: biennial screening for women 40-74."*

### Clinical Knowledge Graph
93 nodes (diagnoses, drugs, complications, interventions) and 67 evidence-based edges sourced from ADA, ACC/AHA, KDIGO, USPSTF, and FDA guidelines. Two-hop traversal finds non-obvious connections: `Hypertension -> Stroke -> Cardiology referral` even when only hypertension is active. Drug-drug interaction detection as a second safety layer.

### FHIR R4 Integration
Dual-mode EHR client: synthetic patients P001-P020 (guaranteed demo data) + live FHIR R4 queries to `hapi.fhir.org/baseR4` for real patients. Uses proper LOINC codes for labs, SNOMED for conditions, RxNorm for medications. Ready to connect to Epic/Cerner with SMART on FHIR OAuth2.

## Production Hardening Notes

Deploying a multi-agent system to free-tier cloud infrastructure (Railway + Streamlit Community Cloud) surfaced several issues worth documenting:

### 1. Memory — 1 GB RAM cap on Railway trial
Initial deploy used sentence-transformers + cross-encoder + torch, consuming ~2.3 GB — instant OOM kills on every agent run. Replaced all local ML models with OpenAI's `text-embedding-3-small` (1536-dim). Removed torch, transformers, and sentence-transformers from `requirements.txt`. Memory now peaks at ~900 MB under load. See [`rag/embedder.py`](rag/embedder.py).

### 2. Single-worker deployment
Running `--workers 2` caused both uvicorn workers to execute the startup seeding hook simultaneously. Both saw an empty ChromaDB, both loaded the embedding model, both started embedding — doubling RAM usage and triggering OOM kills when a workflow request arrived concurrently. Fixed by setting `--workers 1` in [`railway.toml`](railway.toml). Free-tier RAM cannot support two Python processes each holding ChromaDB and agent libraries in memory.

### 3. Ephemeral filesystem
ChromaDB on Railway disk gets wiped on every redeploy. Hardcoded 18 clinical guideline excerpts as Python constants in [`rag/guideline_seed.py`](rag/guideline_seed.py). On startup, if ChromaDB is empty, a background thread embeds the seed set (~30s, ~$0.001) so the first user always has RAG access. The full 63-source scrape is still available on demand via the passcode-gated refresh.

### 4. Token rate limits under concurrent load
gpt-4o-mini has a 200k TPM limit on free tier. A single care gap agent run drains ~180k tokens across its 13-step plan — any second workflow arriving during that window hits 429. Two-layer defense in [`api/main.py`](api/main.py) and [`utils/llm_utils.py`](utils/llm_utils.py):
- **Global workflow lock** — only one workflow runs process-wide; additional requests return HTTP 409 with ETA.
- **Proactive token budget tracker** — sliding 60-second window that mirrors OpenAI's rate-limit algorithm. Every LLM call estimates tokens via `tiktoken`, waits for bucket headroom before calling, then records actual usage from response metadata. Prevents 429s instead of reacting to them.

### 5. Two-mode demo architecture
Rather than letting recruiter traffic drain the OpenAI budget, split the deployment into a **cached read-only layer** (unlimited concurrent users) and a **passcode-gated live layer** (one paid run at a time). See [Live Demo](#live-demo--two-mode-access) above.

## Setup (local development or self-hosting)

```bash
git clone https://github.com/harshinireddy2204/healthcare-ai-agents.git
cd healthcare-ai-agents
python -m venv venv && source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env              # add OPENAI_API_KEY and LANGCHAIN_API_KEY
python -c "from api.main import init_db; init_db()"
```

### First run — populate clinical guidelines

The startup seed embeds 18 core guidelines automatically (~30s, ~$0.001). For the full 63-source catalog:

```bash
python rag/refresh_flow.py        # ~5 minutes, ~$0.10 in embedding tokens
```

### Regenerate the demo run cache

```bash
# With OPENAI_API_KEY set, from the repo root:
python scripts/generate_demo_runs.py          # All 6 patients (~$0.30, ~20 min)
python scripts/generate_demo_runs.py P004     # Just one patient

git add data/demo_runs.json
git commit -m "Regenerate demo runs"
git push
```

The generator pauses 65 seconds between patients so the OpenAI TPM bucket fully refills.

### Start the system

```bash
uvicorn api.main:app --reload --port 8000    # Terminal 1 — API
streamlit run frontend/app.py                # Terminal 2 — Dashboard
python orchestration/prefect_flow.py         # Terminal 3 — Batch scheduler (optional)
```

### Environment variables

| Variable | Required | Purpose |
|---|---|---|
| `OPENAI_API_KEY` | Yes | Agent LLM calls + embeddings |
| `LANGCHAIN_API_KEY` | Optional | LangSmith trace capture |
| `LANGCHAIN_PROJECT` | Optional | LangSmith project name |

## Production Readiness Notes

This is a portfolio/research project using synthetic data. For production deployment at a health system, the following additions would be required:

- **HIPAA BAA** with LLM provider (OpenAI Enterprise or Azure OpenAI)
- **SMART on FHIR OAuth2** for Epic/Cerner authentication
- **De-identification pipeline** (Microsoft Presidio) before LLM API calls
- **PostgreSQL** replacing SQLite for multi-user, audit-compliant storage
- **Clinical validation study** against known-outcome cases
- **IRB approval** for any research involving real patient data

The architecture is designed to support these additions without structural changes.
